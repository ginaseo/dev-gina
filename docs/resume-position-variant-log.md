# 이력서 포지션별 노출 순서 작업 로그

시작부터 지금까지의 의사결정 기록. "무엇을 했는지"보다 "왜 그렇게 했는지"와 트레이드오프에 집중한다. 세부 구현은 [resume-position-variant-design.md](resume-position-variant-design.md), [resume-position-variant-plan.md](resume-position-variant-plan.md) 참고.

## 1. 출발점 — 포지션별로 이력서를 따로 만들 가치가 있는가

백엔드 / AI자동화·AI agent 포지션 지원 시 이력서를 각각 만들어야 하는지가 첫 질문이었다.

**검토한 선택지와 트레이드오프**

| 방향 | 장점 | 단점 |
|---|---|---|
| 완전히 별도 이력서 2~3벌 | 포지션별 콘텐츠 자체를 다르게 쓸 수 있음(강조 문구, 프로젝트 설명 톤 등) | 유지보수 N배, 텍스트 동기화 부담, 실제로는 AI agent 포지션 증거(프로젝트)가 하나뿐이라 콘텐츠 자체를 크게 다르게 쓸 여지가 적음 |
| 콘텐츠는 하나, 노출 순서만 포지션별로 다르게 | 유지보수 하나, 스크리너가 상단만 봐도 관련성 파악 가능 | 포지션별로 "다른 이야기"를 하고 싶을 때는 한계(향후 텍스트 차이 확장 필요) |

**결정**: 순서만 다르게. 실제 병목은 "이력서가 2벌 필요하다"가 아니라 "AI agent 관련 실적 자체가 아직 얇다"는 점이었고, 콘텐츠를 억지로 쪼개는 것보다 순서 제어가 훨씬 싸고 효과가 큼(YAGNI). 텍스트 차이는 "나중에 필요해지면 추가할 확장 포인트"로만 설계에 남기고 지금은 구현하지 않음.

## 2. Variant 선택 메커니즘 — 왜 쿼리 파라미터인가

포지션별 순서를 어떻게 전환할지 3가지를 비교했다.

| 방식 | 비고 |
|---|---|
| URL 쿼리 파라미터 (`?v=backend`) | **채택.** 콘텐츠 소스 하나, 공유 링크로 특정 버전 바로 열림, 인쇄도 같은 DOM이라 자동 대응 |
| 페이지 내 토글 버튼 | 기각. 공유 링크로 특정 포지션용 버전을 못 박음 — 지원 링크를 상대에게 보내는 용도에 안 맞음 |
| 별도 경로 (`/resume/backend/`) | 기각. GitHub Pages는 순수 정적 호스팅이라 서버 라우팅이 없음 → 폴더마다 실제 entry point 파일이 물리적으로 필요, "구조를 하나로 통일"하려는 목적과 충돌 |

## 3. 순서 반영 방식 — 왜 CSS `order`가 아니라 JS가 DOM을 생성하는가

가장 손이 덜 가는 방법은 flex/grid `order` 속성으로 시각적 순서만 바꾸는 것이었다. 하지만 인쇄용 마크업(`.pr-entry`)은 `page-break-inside`/`break-inside` 규칙에 의존하는 순수 block-flow 레이아웃이었고, 이 페이지네이션은 이전 커밋 5개(`Refine print resume pagination...` 등)에 걸쳐 이미 튜닝되어 있었다. 여기에 flex를 얹어 `order`를 쓰면 브라우저별 flex+print 페이지 분할 버그 이력과 겹쳐 기존에 공들인 레이아웃이 깨질 위험이 있었다.

**결정**: JS가 order 배열을 순회하며 실제 DOM을 그 순서로 생성(`appendChild`)하는 방식. DOM 삽입 순서 = 문서 순서이므로 인쇄 CSS는 전혀 건드릴 필요가 없었다. "더 적은 코드"보다 "기존에 이미 튜닝된 무언가를 깨뜨리지 않는 것"을 우선한 판단.

## 4. Codex 리뷰 1차 (설계 문서) — 왜 `PROJECT_META` 스키마가 따로 필요했나

초기 설계는 order 배열(`PROJECT_ORDER`)만 있으면 될 것이라 가정했다. `/codex:adversarial-review`가 지적한 것: 프로젝트별로 실제 노출되는 bullet 개수가 다르다(p1: 2개, p3: 2개, p4: 3개). 게다가 `i18n.js`에는 `p1.li3`, `p3.li3`, `p1.r1/r2` 같은 죽은 키가 존재해서, "키가 있으니 렌더링 대상"이라고 유추하면 없는 콘텐츠를 만들거나 있는 콘텐츠를 빠뜨릴 위험이 있었다.

**결정**: 순서 배열과 별개로 `PROJECT_META`에 프로젝트별 bullet-key 목록을 명시적으로 고정. "존재 여부로 유추" 대신 "선언된 스키마만 신뢰"하는 원칙으로 바꿈 — 이 판단이 이후 구현 전체의 기준이 됨.

## 5. Codex 리뷰 2차 (구현 플랜) — 왜 Task 3가 원자적 커밋이어야 했나

최초 플랜은 "HTML 마크업 제거" 커밋과 "JS 렌더러 연결" 커밋을 분리했다. 이 리포는 GitHub Pages CI가 `main`의 매 커밋을 배포 후보로 다루므로, 그 경계 시점에 merge/배포가 일어나면 실제로 빈 이력서가 나갈 수 있다는 것이 2차 adversarial review의 지적이었다.

**결정**: HTML 제거와 `renderResume()` 연결을 하나의 태스크·하나의 커밋으로 묶음(Task 3). "커밋을 잘게 쪼개는 것"보다 "커밋 경계마다 배포 가능한 상태를 유지하는 것"을 우선. 이후 구현 순서 자체가 Task1(데이터, 죽은 코드)→Task2(템플릿 함수, 죽은 코드)→Task3(원자적 스위치)→Task4(문서)로, 앞의 두 태스크는 의도적으로 "아무것도 호출하지 않는" 안전한 추가만 하도록 설계됨.

## 6. 구현 — subagent-driven-development

4개 태스크를 태스크별 fresh subagent로 구현하고, 태스크마다 별도 리뷰어(spec 준수 + 코드 품질 2단 검증)를 거쳤다. 전 태스크 승인, Minor 지적만 소수(리포트 문구 오류, 미사용 `mode` 파라미터 등 — 코드 정확성과 무관). 이후 opus 모델로 whole-branch 최종 리뷰를 별도로 돌려 브랜치 전체의 원자성 불변식(어느 커밋 경계에서도 이력서가 비지 않음)이 실제로 유지되는지 재검증. 최종 리뷰가 제안한 Minor 하드닝 2건(`PROJECT_META` 불변식 주석, `renderResume()`→`applyLang()` 순서 주석)과 플랜 문서 셀렉터 오류 1건을 fix 서브에이전트로 일괄 반영.

## 7. 병합 및 정리

격리된 git worktree에서 작업 후 `main`에 로컬 merge. 플랜 문서 한 곳에서만 add/add 충돌 발생(같은 파일이 `main`과 feature 브랜치 양쪽에서 독립적으로 "추가"된 형태 — cherry-pick으로 생긴 SHA 불일치가 원인, 내용은 동일해서 feature 브랜치 버전 그대로 채택). Windows 파일 잠금으로 worktree 디렉토리 하나가 디스크에 남았지만 git 상으로는 이미 정리되어 무해.

## 8. 병합 후 발견된 버그 — 다크모드에서 프린트 폰트색이 바뀌던 문제

**증상**: 다크모드 켠 채로 프린트 미리보기를 열면 텍스트 색이 라이트모드 때와 달라짐.

**원인**: `style.css:103`의 `ul { ... color: var(--ink); }`가 스코프 없이 전역으로 걸려 있었다. 인쇄 전용 규칙(`.pr-label-list`, `.pr-entry ul` 등)은 font-size·line-height 같은 레이아웃 속성만 재정의했지 `color`는 건드리지 않아서, 이 전역 규칙이 그대로 새어 들어갔다. `--ink`는 다크모드에서 밝은 값으로 바뀌는 변수라, 흰 배경 인쇄 화면에 거의 안 보이는 밝은 텍스트가 나온 것.

**고친 방식과 이유**: 매 인쇄 규칙마다 `color`를 개별적으로 하드코딩하는 대신, `@media print` 안에서 `--bg`/`--ink`/`--sub`/`--line`/`--card-bg` 변수 자체를 라이트 값으로 되돌렸다(다크 테마 규칙과 동일한 selector 특이도로 작성해 소스 순서로 이기게 함). 개별 패치보다 근본 원인(변수 자체가 테마를 따라간다는 것)을 막는 쪽을 택함 — 앞으로 다른 프린트 규칙이 `var(--ink)`를 새로 써도 같은 버그가 재발하지 않음.

## 10. 다음 작업 후보

### 10-1. Variant별 텍스트 차이 지원 (보류 중인 확장 포인트)

설계 당시부터 "지금은 순서만, 텍스트는 나중에"로 미뤄둔 부분. 필요해지면 i18n 키 조회 시 `projects.p1.li1__backend` 같은 variant suffix를 우선 시도하는 fallback만 추가하면 됨 — 설계 문서에 컨벤션만 남겨두고 아직 코드에는 반영하지 않음.

### 10-2. Timeline ↔ Resume 구조 통합 (신규 후보, 아직 스펙 없음)

사용자가 아래 검토안을 제시함 — 그대로 인용·정리:

**장단점 비교**

| 장점 | 단점 |
|---|---|
| 데이터를 한 곳에서 관리 가능 | 초기 리팩터링 규모가 커짐 |
| 프로젝트를 한 번 수정하면 Timeline과 Resume 모두 반영 | Timeline과 Resume 요구사항이 달라질 경우 공통 구조가 오히려 복잡해질 수 있음 |
| 중복 HTML/JS 감소 | 공통 구조를 잘못 설계하면 두 화면 모두 영향을 받음 |
| Variant(master/backend/ai) 적용이 쉬움 | 작은 변경도 공통 코드 검토가 필요 |
| 유지보수 비용 감소 | 초기 설계 비용 증가 |
| 프로젝트 추가/삭제가 쉬움 | 데이터 모델 설계를 신중하게 해야 함 |

**제안된 방향**: Timeline과 Resume는 UI는 달라도 데이터는 같아야 한다.

- **공통화할 것**: 프로젝트 데이터, 기술스택 데이터, Variant 순서, 다국어 문자열
- **분리할 것**: HTML 레이아웃, CSS, 화면별 렌더 함수

예: `projects.js` 하나를 공통 소스로 두고, Resume Renderer는 이력서 형태로, Timeline Renderer는 타임라인 형태로 각자 출력.

**검토 의견 (이 문서 작성 시점 기준)**: 방향 자체는 동의 — "UI는 다르고 데이터는 하나"는 이번 작업에서 이미 적용한 원칙(순서 배열 + 스키마를 데이터로, 렌더링만 화면/인쇄로 분기)과 같은 결이라 자연스러운 확장이다. 다만 이 저장소에는 위 비교표에 없는 구체적 장애물이 하나 있다: **`site-root/`는 빌드 단계가 전혀 없는 순수 정적 파일**(AGENTS.md 명시 원칙)인 반면, **`src/`(Timeline)는 Vite + TypeScript + React 빌드 산출물**이다. "공통 `projects.js` 하나"를 만들려면 최소한 다음을 결정해야 함:
- 데이터 포맷: 순수 JS 전역 변수(`var`, 지금 `i18n.js` 방식)로 둘 다 읽게 할지, JSON으로 두고 양쪽 다 `fetch`로 읽을지, 아니면 TS 쪽만 `import`하고 정적 사이트는 별도로 읽을지
- 배치 위치: 두 앱이 배포 시 같은 오리진(`/dev-gina/`) 아래 있다는 점(site-root → pages 루트, Timeline → `pages/timeline/`)을 활용해 `/dev-gina/shared/...` 같은 공용 경로를 둘지
- `deploy.yml` 변경 필요 여부: 지금은 `dist/`와 `site-root/`를 각각 복사만 하는데, 공유 데이터 파일을 두 산출물 어디에 어떻게 포함시킬지

이 정도 규모의 아키텍처 변경은 바로 설계 문서로 넘어가기보다 **별도 브레인스토밍 세션**(요구사항·제약 재확인, 2~3안 비교)을 먼저 거치는 게 맞다고 판단 — 이번 항목은 "다음에 다룰 후보"로만 기록.
